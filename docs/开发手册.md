# STF 中文开发手册

本文面向日常开发与实验迭代，目标是：
- 改动最小
- 复用最大
- 故障定位成本最低

## 1. 先理解当前运行链路

运行链路固定为：
1. CLI 读取配置文件：`stf/cli/main.py`
2. 配置加载为 `ExperimentConfig`：`stf/config/loader.py`
3. 引擎执行训练/评估/推理：`stf/engine/*.py`
4. 日志、checkpoint、图像写入 `runs/<task>/<exp_timestamp>/...`

你需要优先保证“配置可构建 + 模型接口兼容”，再考虑改引擎。

## 2. 一次完整实验的标准流程

1. 安装环境
```bash
uv sync --group dev
uv pip install torch torchvision --index-url https://download.pytorch.org/whl/cu124
```

2. 准备数据路径（例如复用旧仓库数据）
```bash
ln -s /home/hang/repos/stf-tmp/data ./data
```

3. 选一个最小配置验证链路
```bash
uv run stf train --config configs/flow/minimal.py
```

4. 看输出是否完整
- `checkpoints/model_epoch_*.pth`
- `logs/train.log`
- `tensorboard/events...`

5. 再逐步替换你的模型/数据/指标

## 3. 配置自定义指南（最高性价比）

配置入口统一为：
- `EXPERIMENT = ExperimentConfig(...)`

关键字段：
- `task`: `flow` / `stfdiff`
- `name`: 实验名
- `data`: `DataConfig(train_dataloader, val_dataloader, test_dataloader)`
- `model`: 模型实例
- `optimizer`: 优化器构造器
- `scheduler`: 调度器构造器（可选）
- `metrics`: 指标列表
- `train`: `TrainConfig(...)`
- `io`: `IOConfig(...)`
- `resume_from`: checkpoint 路径（可选）

建议：
- 把所有实验变量都放在配置里（学习率、batch size、timesteps、loss_type 等）。
- 尽量不要把实验常量硬编码进 `engine`。

## 4. “最小改动、最强复用”的开发策略

### 4.1 只调参（推荐）

适用：学习率、batch size、epoch、损失超参变动。

改动位置：
- 仅 `configs/*.py`

影响面：
- 最小

### 4.2 替换模型（推荐）

适用：改网络结构或算法细节。

改动位置：
- `stf/models/<your_model>.py`
- `stf/models/__init__.py`（导出）
- `configs/*.py`（替换 `model=...`）

复用关键：
- 保持与引擎兼容接口：
  - 训练：`forward(coarse_img_01, coarse_img_02, fine_img_01, fine_img_02)` 返回标量 loss
  - 推理：`sample(coarse_img_01, coarse_img_02, fine_img_01)` 返回预测图像

如果新模型接口不同：
- 优先写适配器（wrapper）对齐接口
- 尽量不改 `stf/engine/train.py`

### 4.3 增加数据变换（推荐）

适用：新归一化、增强、预处理。

改动位置：
- `stf/data/transforms/<new_transform>.py`
- `stf/data/transforms/__init__.py`
- `configs/*.py` 中 `transform_func_list`

影响面：
- 中等，通常不影响训练主循环

### 4.4 增加指标（推荐）

适用：新增评价指标。

改动位置：
- `stf/metrics/<new_metric>.py`
- `stf/metrics/__init__.py`
- `configs/*.py` 的 `metrics=[...]`

注意：
- 指标类需有 `__name__`（用于日志命名）
- 指标 `forward` 输入遵循 BCHW

### 4.5 修改引擎（最后手段）

适用：确实需要改变训练流程本身（多优化器、梯度累积策略改变等）。

改动位置：
- `stf/engine/base.py`
- `stf/engine/train.py`
- `stf/engine/eval.py`

风险：
- 影响面最大，容易破坏现有任务。

建议：
- 先通过配置和模型适配器解决 80% 需求。

## 5. 模块增减“改哪里最划算”速查表

| 目标 | 最少改动文件 | 可选改动 | 复用建议 |
|---|---|---|---|
| 调参 | `configs/*.py` | 无 | 通过配置分离实验变量 |
| 换模型 | `stf/models/*`, `stf/models/__init__.py`, `configs/*.py` | `stf/compat/*` | 保持 `forward/sample` 签名不变 |
| 增指标 | `stf/metrics/*`, `stf/metrics/__init__.py`, `configs/*.py` | 无 | 指标实例只在配置注册 |
| 增数据处理 | `stf/data/transforms/*`, `__init__.py`, `configs/*.py` | `stf/data/dataset.py` | 尽量做成无状态 transform |
| 接旧配置 | `stf/compat/migration.py` | `docs/migration.md` | 不侵入核心引擎 |
| 接旧权重 | `stf/compat/checkpoint.py` | `stf/engine/*.py` | 兼容层集中管理 key 映射 |

## 6. 旧项目迁移建议

### 6.1 迁移旧配置

```bash
uv run stf migrate-config \
  --legacy-config /home/hang/repos/stf-tmp/config/flow/toy_train_config.py \
  --output configs/flow/migrated_toy.py
```

### 6.2 迁移旧 checkpoint

引擎已通过 `stf.compat.load_legacy_checkpoint` 兼容以下 key：
- `model`
- `optimizer`
- `ema`

并支持从 `model_epoch_<n>.pth` 推断 epoch。

## 7. 开发前后最小检查清单

开发前：
- 配置是否能成功 import
- 数据路径是否有效

开发后：
- `uv run pytest -q tests/smoke`
- `uv run stf --help`
- 至少跑一次 toy 训练

## 8. 常见问题

1. 报 `train_dataloader is required`
- 配置里没有设置 `data.train_dataloader`。

2. 报 CUDA 不可用
- 使用 `uv run` 执行并检查：
```bash
uv run python -c "import torch; print(torch.cuda.is_available())"
```

3. 验证阶段 `sample` 报错
- 模型没有实现 `sample(...)` 或参数签名与引擎预期不一致。
- 优先用模型适配器对齐签名。
